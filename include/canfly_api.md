# CanFly OS API Reference

**Version:** 1.0  
**Target:** pic32mz

## Overview

The CanFly operating system provides a user-mode/kernel-mode split architecture
for the PIC32MZ (MIPS microAptiv) processor. User-mode applications interact
with the kernel exclusively through system calls using the MIPS `SYSCALL`
instruction.

The SDK consists of:
- **canfly.h** — Public header with all function declarations
- **canfly_stubs.c** — SYSCALL stub implementations (link with your app)
- **canfly_user.c** — User-mode library (dispatch_message, defwndproc, etc.)
- **Proton widget library** — Open-source widget implementations

### Architecture

```
┌──────────────────────────────────────────────┐
│  User Application (generated by yehudi)      │
│  + Proton Widgets + canfly_user.c            │
├──────────────────────────────────────────────┤
│  canfly_stubs.c  (SYSCALL inline asm)         │
╠══════════════════════════════════════════════╣ ← SYSCALL boundary
│  Kernel Dispatch (atom_dispatch.c)           │
├──────────────────────────────────────────────┤
│  Atom (hardware abstraction)                 │
│  Neutron (kernel + config DB)                │
│  Photon (2D GDI)                             │
│  Map Renderer                                │
└──────────────────────────────────────────────┘
```

### Naming

| Layer | Description |
|-------|-------------|
| **CanFly** | The OS — the published SDK and syscall interface |
| **Atom** | Internal hardware abstraction (not published) |
| **Neutron** | Kernel: scheduling, config DB, CAN routing, streams |
| **Photon** | 2D graphics device interface |
| **Proton** | Widget library (open source, runs in user mode) |

### Message-Driven Architecture

The kernel **never** calls back into user space. All notifications from kernel
to application are via CAN messages posted to the window message queue:

- `id_paint` (1428) — window needs repaint
- `id_create` (1429) — window created
- `id_map_render_complete` — map canvas ready to blit
- Subscribed CAN bus messages (sensor data, autopilot, etc.)

### CAN Message Subscription

Applications subscribe to blocks of CAN message IDs using `subscribe()`.
Each task supports up to 8 subscription blocks.

| Block | ID Range | Description |
|-------|----------|-------------|
| Alarms | 0–255 | Engine/system alarms |
| AHRS | 256–319 | Attitude, heading, airspeed, altitude |
| EDU | 320–447 | Engine data unit |
| Left kMAG | 448–479 | Left magneto |
| Right kMAG | 480–511 | Right magneto |
| Autopilot | 512–627 | AP, navigation, flight plan |
| Servos | 628–819 | Roll, pitch, yaw servos |
| Status | 1350–1399 | Node status and boot |

**Reserved ranges (cannot be subscribed):**
- 1400–1499: Local window events (always delivered)
- 1500–2047: Neutron pipes and services

### `sys_subscribe`

Subscribe to a block of CAN message IDs. Messages with IDs in
        the range [first_id, last_id] inclusive will be posted to the
        specified window's message queue.

        Restrictions:
        - first_id must be less than or equal to last_id
        - Neither ID may be in the range 1500..2047 (reserved for pipes)
        - Maximum 8 subscription blocks per task

        Returns a handle that can be closed to unsubscribe.

**Syscall ID:** 6

**Kernel function:** `subscribe_block`

```c
result_t sys_subscribe(uint16_t first_id, uint16_t last_id, handle_t hwnd, handle_t* hndl);
```

## User-Mode Library Functions

These functions are **not** syscalls. They execute entirely in user space
and are part of the CanFLy SDK link library.

### `dispatch_message`

Retrieves the wndproc via `get_wndproc` (syscall) and calls it in user space.

### `defwndproc`

Default window message handler. On `id_paint`: calls `begin_paint` (syscall),
processes paint, calls `end_paint` (syscall). All in user space.

### `send_message`

Synchronous message dispatch. Calls wndproc directly without the queue.

### `is_dialog_message`

Tab chain navigation for dialog-style layouts.
